The Social Q&A Platform differs significantly from the BookStore example because it is centered around user interaction instead of product sales. Users create posts, comment on discussions, vote on both posts and comments, and follow content to receive updates. These interactions generate events that are processed asynchronously by the Notification service. Unlike a bookstore, there is no inventory, checkout process, payment processing, or order lifecycle. Instead, the application's core business logic focuses on community engagement, content ranking through voting, and real-time notifications.

Technology Stack Justification
Programming Languages
Go (Posts & Comments Service): Go provides high performance and excellent concurrency, making it well suited for handling large numbers of read-heavy requests for posts and comments.
Node.js with Express (User & Authentication Service): Express provides rapid API development along with mature JWT authentication libraries. Since authentication is primarily I/O bound, Node.js is an excellent fit.
Python (Notification Service): Python simplifies background workers and event consumers, making it a good choice for processing Kafka events without affecting user response times.
Database Justification

PostgreSQL

PostgreSQL is used by every service because the application's data is highly relational.

Examples include:

Users own Posts
Posts contain Comments
Users Follow Posts
Users Vote on Posts and Comments

These relationships benefit from foreign keys, transactions, and ACID guarantees. The schema is relatively stable, so there is little advantage to using a NoSQL database.

Each microservice owns its own PostgreSQL database to maintain proper service boundaries.

Redis

Redis is only used by the Authentication service.

Authentication data such as refresh tokens and active sessions are temporary, making Redis an ideal choice because it provides:

extremely fast lookups
in-memory performance
automatic expiration using TTL

If Redis becomes unavailable, users simply authenticate again since no permanent application data is lost.

Kafka

Kafka is used for asynchronous communication between services.

Instead of making synchronous API calls for every notification, services publish events such as:

New Post Created
New Comment Added
Post Followed

The Notification service consumes these events and creates notifications for users following the relevant content. This keeps services loosely coupled and allows notification processing to scale independently from the core application.

Section 5:
Constraints & Assumptions
Assumptions
The application is deployed in a single geographic region.
Docker Compose is used for deployment rather than Kubernetes.
Every microservice owns its own PostgreSQL database.
Authentication uses JWT access tokens with refresh tokens stored in Redis.
Only authenticated users may create posts, comments, vote, or follow content.
Email verification and password recovery are outside the scope of this project.
Constraints
PostgreSQL is used throughout the project to simplify deployment while maintaining relational integrity.
Services communicate synchronously using REST APIs whenever an immediate response is required.
Kafka is used for asynchronous event processing so notification generation does not increase request latency.
Redis stores only temporary authentication data and is not used for permanent storage.
High availability, automatic failover, and multi-region deployment are outside the scope of this assignment.
Key Design Decisions
Every service owns its own database to preserve microservice independence.
REST APIs are used for direct client and service communication because they are simple to implement and document.
Kafka enables event-driven communication between services, reducing coupling and improving scalability.
PostgreSQL was selected because the application's entities have strong relationships that benefit from relational modeling and transactions.
Redis is used exclusively for authentication because session data is temporary and benefits from fast in-memory storage.
